# プロジェクト構成の説明

このドキュメントでは、**なぜこのディレクトリ構成にしているのか** をフォルダ単位で解説します。
データ分析の現場でよく使われる構成 (Cookiecutter Data Science など) を参考に、教材向けに簡略化したものです。

---

## 全体像

```
data_analysis_project/
├── data/        … "材料"
├── notebooks/   … "実験ノート" (試行錯誤)
├── src/         … "確定したレシピ" (再利用ロジック)
├── models/      … "完成品"
├── docs/        … "説明書"
├── main.py      … "ボタンひとつで一連の調理を行う装置"
└── requirements.txt
```

料理に例えると上のようなイメージです。
それぞれの役割を見ていきます。

---

## 1. `data/` — データの置き場

```
data/
├── raw/         # 生データ (一切加工していないもの)
└── processed/   # 前処理・特徴量生成後のデータ
```

### なぜ `raw` と `processed` を分けるのか

- **再現性のため**: いつでも `raw` に戻って同じ加工を再実行できる
- **誤って書き換えても安全**: `processed` を再生成すれば良いだけ
- **チームでも混乱しない**: 「これは加工済み?」がディレクトリ名で分かる

### `.gitkeep` ファイル

空ディレクトリは Git で管理できないので、ダミーファイル `.gitkeep` を置いて
ディレクトリの存在だけは追跡します (中身のデータは `.gitignore` で除外)。

### `data/` を Git に入れない理由

- データはサイズが大きい → リポジトリが重くなる
- 個人情報を含むことがある → 漏洩リスク
- ロジックが再現できれば、データは何度でも生成できる

---

## 2. `notebooks/` — 試行錯誤の場

Jupyter Notebook (`.ipynb`) を置く場所。
**「読み物としての分析レポート」「探索的なメモ」** を担当します。

```
notebooks/
├── 01_EDA.ipynb
├── 02_preprocessing.ipynb
├── 03_modeling.ipynb
└── 04_evaluation.ipynb
```

### 番号付けの理由

ファイル名の先頭に番号を付けることで、

- 実行順が一目で分かる
- ファイルブラウザでの並び順も実行順になる

### ノートブックで重要な原則

> **「ロジックの本体は notebook に置かない」**

ノートブックの中で長い関数や複雑なクラスを定義すると…

- 他のノートブックから再利用できない
- 差分レビューが大変 (JSON 全体が変わる)
- バグを直しても他に伝播しない

**「実験はノートブックで, 確定したコードは `src/` の .py に移す」** が鉄則です。

---

## 3. `src/` — 再利用ロジックの本体

ノートブックから `from src import xxx` の形で呼び出される **モジュール群**。

```
src/
├── __init__.py        # パッケージ宣言
├── data_loader.py     # データの取得と保存
├── preprocessing.py   # 欠損値処理・スケーリング
├── features.py        # 特徴量生成
├── modeling.py        # 学習・保存・読み込み
└── evaluation.py      # 評価指標
```

### 機能ごとにファイルを分ける理由

- **凝集度が上がる**: 「欠損値の処理を直したい」ときに `preprocessing.py` だけ見れば良い
- **テストしやすい**: 関数単位で動作確認できる
- **チーム分業が可能**: 担当ファイルを分けられる

### `__init__.py` の役割

ディレクトリを「Python パッケージ」として認識させるためのファイル。
`from src import data_loader` という import が書けるようになります。

---

## 4. `models/` — 学習済みモデルの保管

`pickle` 形式で学習済みモデルを保存します (例: `linear_regression.pkl`)。

### なぜ保存するのか

- **再学習は時間がかかる**: 大きなデータだと数時間〜数日かかる
- **デプロイで使い回せる**: 学習はオフラインで、推論だけ別環境で実行
- **再現性**: 「あの時のモデル」を再現できる

### バージョン管理のコツ

ファイル名に日時やバージョンを入れると履歴を追いやすい:

```
linear_regression_v1.pkl
linear_regression_20260526.pkl
```

---

## 5. `main.py` — 一括実行のエントリポイント

ノートブックで個別に検証した処理を、`main.py` から **コマンドラインで一気に実行** できるようにしておきます。

```bash
python main.py
python main.py --model xgboost
```

### メリット

- バッチ処理 (毎日自動実行など) に組み込める
- 「ノートブックを順に開いて Shift+Enter を連打」しなくても完結
- 引数で挙動を切り替えられる (モデルの種類, ハイパーパラメータなど)

## 6. `requirements.txt` — 依存ライブラリの記録

```
numpy
pandas
scikit-learn
matplotlib
seaborn
japanize-matplotlib
jupyter
xgboost
```

### なぜ重要か

- **同じ環境を再現できる**: `pip install -r requirements.txt` で揃う
- **バージョンずれによるバグを防げる** (`pandas==2.0.3` のように固定するとなお良い)
- **チーム共有が楽**

## まとめ: なぜわざわざ分割するのか

「単一の ipynb」で動かしているうちは便利ですが、規模が大きくなったり、
チーム開発になったりすると以下の問題に必ずぶつかります。

| 単一 ipynb で起きる問題 | この構成での対処 |
|-|-|
| 同じ前処理を別ノートでも書く羽目になる | `src/preprocessing.py` に集約して再利用 |
| Git レビューで差分が読めない | `.py` 化することで行単位で差分が出る |
| ファイルが重くて開けない | データやモデルは `data/`, `models/` に分離 |
| 「これは古いコード?」「これは加工済み?」が分からない | ディレクトリ名と命名規則で明示 |
| 一人で全部抱え込みがち | モジュール単位で担当を分けられる |

**「`src/` に確定ロジックを置く・`notebooks/` で試行錯誤する」** という分担を意識すると、
スムーズに移行できます。
